# Semanas 1 e 2 — Segurança e Pipeline Editorial

## Objetivo

Implementar pipeline editorial seguro onde nenhum conteúdo é publicado sem passar por:
ficha de pesquisa → geração de rascunho → validação → Pull Request → aprovação → publicação

**Métrica alvo:** 0 publicações não revisadas

---

## Tarefas

### S1-01: Proteger branch `main`

**Responsável:** Tecnologia
**Prioridade:** P0 — Bloqueador
**Risco:** Crítico — publicação não autorizada
**Dependências:** Nenhuma

**Descrição:**
Configurar proteção da branch `main` no GitHub para impedir pushes diretos. Toda alteração deve passar por Pull Request com validação.

**Critérios de aceite:**
- [ ] Push direto para `main` é rejeitado pelo GitHub
- [ ] Pull Request é obrigatório para qualquer alteração
- [ ] Pelo menos 1 aprovação de review é necessária
- [ ] Status checks devem passar antes do merge
- [ ] Branch fica atualizada com `main` antes do merge

**Evidência de conclusão:**
Print da configuração de branch protection OU link para as regras da branch.

### S1-02: GitHub Actions — Validação automática em PR

**Responsável:** Tecnologia
**Prioridade:** P0 — Bloqueador
**Risco:** Crítico — conteúdo não validado entra em produção
**Dependências:** S1-01

**Descrição:**
Criar workflow do GitHub Actions que executa automaticamente em todo PR:
1. Validação de schemas de dados (`validate-all.js`)
2. Validação de frontmatter dos posts (`validate_posts.js`)
3. Verificação de fontes existentes
4. Verificação de links funcionando
5. Build do Jekyll

**Critérios de aceite:**
- [ ] Workflow é executado automaticamente em todo PR
- [ ] Falha de validação impede o merge
- [ ] Resultado é visível na PR (status check)
- [ ] Logs de erro são claros e acionáveis

**Evidência de conclusão:**
Workflow YAML funcionando, status checks aparecendo em PRs.

### S1-04: Remover token da URL do Git

**Responsável:** Tecnologia
**Prioridade:** P0 — Bloqueador
**Risco:** Crítico — exposição de credenciais
**Dependências:** Nenhuma

**Descrição:**
Verificar se o GITHUB_TOKEN aparece em URLs de remote, logs, ou dump de variáveis. Garantir que tokens são usados apenas via header `Authorization`.

**Verificação:**
- `git remote -v` contém token na URL?
- Algum script loga a URL completa com token?
- Algum arquivo `.env` ou `.env.example` expõe token em URL?

**Critérios de aceite:**
- [ ] Nenhum remote do Git contém token na URL
- [ ] Nenhum log ou saída de terminal exibe o token
- [ ] `.env.example` documenta apenas o placeholder do token
- [ ] Token é usado exclusivamente via header `Authorization: Bearer`

**Evidência de conclusão:**
`git remote -v` mostrando URLs sem token; revisão de todos os scripts de log.
